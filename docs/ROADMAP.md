## Phase 1: 프로젝트 골격 (1~2일)

### 왜 이 순서인가?

프로젝트 구조와 환경 설정은 모든 개발의 전제 조건이다. 디렉토리 구조, 패키지, 환경 변수가 먼저 갖춰져야 이후 코드가 어디에 위치해야 하는지, 어떤 도구를 사용할 수 있는지가 명확해진다. 골격 없이 기능부터 만들면 나중에 구조를 뜯어고치는 비용이 발생한다.

### 작업 목록

- [ ] Next.js 15 + TypeScript 프로젝트 생성
- [ ] Tailwind CSS, shadcn/ui, Lucide React 설치 및 설정
- [ ] `@notionhq/client` 패키지 설치
- [ ] 환경 변수 설정 (`.env.local`)
  - `NOTION_API_KEY`
  - `NOTION_DATABASE_ID`
- [ ] 기본 디렉토리 구조 설정
  ```
  src/
  ├── app/              # Next.js App Router
  ├── components/       # 공통 컴포넌트
  ├── lib/              # Notion API 유틸리티
  └── types/            # 공통 타입 정의
  ```
- [ ] Notion 데이터베이스 생성 및 API 키 발급
  - Title, Category, Tags, Published, Status 필드 구성
- [ ] ESLint, TypeScript strict 모드 설정

### 완료 기준

- 프로젝트가 로컬에서 정상 실행된다 (`npm run dev`)
- Notion API 연결 테스트가 통과한다
- 환경 변수가 `.env.local`에 올바르게 설정되어 있다

## Phase 2: 공통 모듈 (2~3일)

### 왜 이 순서인가?

Notion API 함수, 타입, 공통 컴포넌트는 Phase 3~4의 모든 기능이 의존하는 코드다. 이것들이 없으면 기능 페이지를 만들 때마다 같은 코드를 반복하거나, 타입이 맞지 않아 수정이 연쇄적으로 발생한다. 공통 모듈을 먼저 완성하면 이후 단계에서 조립하는 방식으로 빠르게 진행할 수 있다.

### 작업 목록

#### Notion API 공통 함수 (`src/lib/notion.ts`)
- [ ] `fetchPages()` - 발행된 글 목록 조회 (Status = 발행됨, 발행일 내림차순)
- [ ] `fetchPageContent(pageId)` - 특정 글의 블록 콘텐츠 조회
- [ ] `fetchCategories()` - 카테고리 목록 동적 조회

#### 공통 타입 정의 (`src/types/`)
- [ ] `Post` 타입 - 제목, 카테고리, 태그, 발행일, 상태
- [ ] `Category` 타입 - 카테고리명
- [ ] `NotionBlock` 타입 - 블록 렌더링용 타입

#### 공통 컴포넌트 (`src/components/`)
- [ ] `Header` - 블로그 헤더 (로고, 네비게이션)
- [ ] `Footer` - 블로그 푸터
- [ ] `PostCard` - 글 목록 카드 (제목, 카테고리, 태그, 발행일)
- [ ] 기본 레이아웃 (`src/app/layout.tsx`) - Header, Footer 포함

### 완료 기준

- Notion API 함수 호출 시 정상적으로 데이터를 반환한다
- 공통 타입이 `any` 없이 TypeScript strict 모드를 통과한다
- Header, Footer 컴포넌트가 반응형으로 동작한다

## Phase 3: 핵심 기능 (3~4일)

### 왜 이 순서인가?

글 목록과 글 상세 페이지는 블로그의 존재 이유 그 자체다. 이것이 없으면 블로그가 아니다. 카테고리 필터나 검색은 없어도 블로그가 동작하지만, 글을 읽을 수 없으면 어떤 부가 기능도 의미가 없다. 가장 핵심적인 사용자 흐름(글 목록 → 글 읽기)을 먼저 완성해야 이후 기능의 가치도 판단할 수 있다.

### 작업 목록

#### 글 목록 페이지 (`src/app/page.tsx`)
- [ ] `fetchPages()`를 활용한 발행된 글 목록 렌더링
- [ ] `PostCard` 컴포넌트로 제목, 카테고리, 태그, 발행일 표시
- [ ] 발행일 기준 내림차순 정렬 적용
- [ ] ISR(Incremental Static Regeneration) 또는 SSG 적용

#### 글 상세 페이지 (`src/app/posts/[id]/page.tsx`)
- [ ] `fetchPageContent(pageId)`로 Notion 페이지 블록 조회
- [ ] 글 상단 메타 정보 표시 (제목, 카테고리, 태그, 발행일)
- [ ] 이전/다음 글 이동 링크

#### Notion 블록 렌더러 (`src/components/NotionRenderer.tsx`)
- [ ] 단락 (paragraph)
- [ ] 제목 H1, H2, H3 (heading_1, heading_2, heading_3)
- [ ] 코드 블록 (code) - 언어 표시 포함
- [ ] 이미지 (image) - alt 속성 필수
- [ ] 인용 (quote)
- [ ] 순서 있는 목록 (numbered_list_item)
- [ ] 순서 없는 목록 (bulleted_list_item)
- [ ] 구분선 (divider)

### 완료 기준

- 홈(`/`)에서 발행된 글 목록이 정상 출력된다
- `/posts/[id]`에서 Notion 본문이 올바르게 렌더링된다
- 지원하는 모든 블록 타입이 정상 표시된다

## Phase 4: 추가 기능 (2~3일)

### 왜 이 순서인가?

카테고리 필터링, 검색, SEO는 핵심 기능 위에 쌓이는 편의 기능이다. 글 목록이 없으면 필터링할 대상이 없고, 글이 없으면 검색할 콘텐츠도 없다. 핵심 흐름이 완성된 뒤에야 "어떤 글이 많은지", "사용자가 어떻게 탐색하는지"가 보이고, 그때 추가 기능의 방향도 자연스럽게 결정된다.

### 작업 목록

#### 카테고리 필터링
- [ ] 카테고리 필터 탭 UI 구현 (전체 / 카테고리별)
- [ ] 카테고리 선택 시 해당 글만 필터링
- [ ] 카테고리 페이지 (`/category/[category]`) 구현
  - 카테고리 헤더 및 글 개수 표시

#### 검색 기능
- [ ] 검색 입력창 UI 구현
- [ ] 제목 및 태그 기준 클라이언트 사이드 실시간 필터링

#### SEO 최적화
- [ ] 각 페이지 `metadata` 설정 (title, description)
- [ ] 글 상세 페이지 Open Graph 메타태그 설정
- [ ] 시맨틱 HTML 태그 적용 확인

### 완료 기준

- 카테고리 탭 클릭 시 해당 글만 표시된다
- 검색어 입력 시 실시간으로 결과가 필터링된다
- `/category/[category]` 페이지가 정상 동작한다
- 각 페이지 SEO 메타태그가 올바르게 설정된다

## Phase 5: 최적화 및 배포 (1~2일)

### 왜 이 순서인가?

최적화는 동작하는 것이 있어야 측정할 수 있다. 기능도 없는 상태에서 성능을 잡으면 방향을 잘못 잡거나 불필요한 작업을 하게 된다. 모든 기능이 완성된 뒤 실제 동작을 기준으로 병목을 찾고, 마지막으로 배포하여 완성된 결과물을 외부에 공개한다.

### 작업 목록

#### 성능 최적화
- [ ] ISR revalidate 주기 설정 (글 목록, 상세 페이지)
- [ ] Next.js `Image` 컴포넌트로 이미지 최적화
- [ ] 불필요한 리렌더링 점검

#### 반응형 디자인 개선
- [ ] 모바일(375px) 레이아웃 최종 점검
- [ ] 태블릿(768px) 레이아웃 최종 점검
- [ ] 데스크탑(1280px) 레이아웃 최종 점검

#### Vercel 배포
- [ ] Vercel 프로젝트 생성 및 GitHub 연동
- [ ] 환경 변수 설정 (`NOTION_API_KEY`, `NOTION_DATABASE_ID`)
- [ ] 배포 후 전체 기능 동작 확인

### 완료 기준

- Vercel 배포 URL에서 전체 기능이 정상 동작한다
- Lighthouse 성능 점수 90점 이상
- 모바일, 태블릿, 데스크탑에서 레이아웃이 깨지지 않는다
- Notion API 키가 클라이언트에 노출되지 않는다
